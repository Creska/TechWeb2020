CONCEPT PLAYER-EDITOR:
* Il player avrà un foglio CSS opportuno. Tutti gli elementi del player hanno nomi standard come: "class='AnswerField' id='Quest1Activity1AnswerField'" e via dicendo. Oppure per il caso dei ligi si può utilizzare la notazione gerarchica di CSS (#Quest0 > #Activity0 per esempio)
* ESEMPIO DI STRUTTURA - con i nomi delle classi:
    * Quest
        * Activity
            * ActivityParagraph
            * AnswerField
        * Activity ...

ROBA DA FARE IMPORTANTE:
* EDITOR
    * Controllare la faccenda della traduzione di apici e quotes
    * Usare cssstylesheet.insertrule() per aggiungere regole vuote al css - sempre che funzioni
    * Aggiungere una guida all'editor css. Scrivere che esistono i container attività e quest - spiegare come modificare gli elementi (soprattutto i figli di attività e quest). Mettere l'elenco delle classi modificabili e ricordare dell'esistenza del selettore nth-child
    * Completare l'astrazione del file system (creare un file css automaticamente ogni volta che si crea una nuova storia)
    * Nella pubblicazione magari mettere un ciclo che controlla se sia presente la risposta giusta nel caso non sia spuntata NEEDEVAL, che non ci siano attività o quest vuote, e robe simili
    * Per comodità dell'utente, aprire l'editor CSS in una nuova scheda
    * Meglio mettere un input del nome della storia al momento della creazione, per gestire al meglio il collegamento col file css
* EDITOR E PLAYER
    * Creare una nuova funzione per fare un reloading dell'attività nel player (nel caso la risposta sbagliata causi la ripetizione della suddetta attività)
